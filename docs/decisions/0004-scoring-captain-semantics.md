# 0004 — Scoring values and captain/vice-captain semantics

## Context

PLAN.md section 6 and AGENTS.md 11.2/11.12 specify an exact fantasy scoring table and a captain multiplier that must apply uniformly to every ledger row, including reversals, not just positive events.

## Decision

- Scoring values are fixed constants in `packages/scoring/src/index.ts` (`STARTING_APPEARANCE +1`, `GOAL` GK/DEF +6, MID +5, FWD +4, `YELLOW_CARD -1`, `DIRECT_RED_CARD -3`, `CLEAN_SHEET` GK/DEF +4, MID +1, FWD 0, `GOALS_CONCEDED -1`, etc.) and are not configurable per contest — one scoring version (`v1.0.0`) applies to every contest until a new version is explicitly cut.
- The captain multiplier (`×2`) is applied once, at ledger-row creation time, to `basePoints` to produce `appliedPoints` (`scoreAction` in `packages/scoring/src/index.ts`). Because reversal rows are generated by negating an already-multiplied `appliedPoints` (`reverse()`), the multiplier is preserved automatically on reversal without a second multiplication.
- Vice-captain fallback is a final-replay-only decision, never applied during live/provisional scoring: if the captain's rebuilt participation interval (`activeSeconds`) is exactly zero at full replay, the vice-captain's already-recorded base ledger rows are re-emitted once more (giving them an effective ×2), via `VICE_CAPTAIN_FALLBACK`. One second of captain participation permanently disables fallback for that entry.

## Alternatives

Apply the captain multiplier at read time (sum base points, then multiply the total) — rejected because it breaks the append-only ledger invariant (`entry_total = sum(applied_points)`) and makes reversal rows ambiguous about whether they were already multiplied.

## Consequences

`fantasy_ledger.applied_points` is always the final, already-multiplied, sign-correct value. Nothing downstream (leaderboard, Telegram messages, worker) re-applies a captain multiplier — doing so would double it.

## Migration

A future scoring version bump must ship as a new `scoringVersion` string, not an in-place edit to the existing point table, and must not retroactively alter locked teams' `scoring_version` field.
