# 0013 — Automated contest lifecycle (devnet)

## Context

Running a contest required a human to run `preflight-fantasy-contest.mjs --send` locally, then manually update six `NEXT_PUBLIC_FANTASY_*` env vars and redeploy the frontend, then after the match manually build a payout tree and run `preflight-fantasy-settlement.mjs --send`. There was also a latent bug: `frontend/app/api/contest/lock-team/route.ts` hardcoded `fixtureId = "18175981"` in source, independent of the env vars, so a second contest couldn't actually be entered correctly even with the env vars updated.

The user asked for this to be fully automated — new contests auto-created for new TxLINE fixtures, live scoring running concurrently across fixtures, settlement auto-published after each match ends, zero manual steps per contest.

ADR 0008 states its single-authority-key settlement trust model "must not be reused... without adding multi-signer publication and/or on-chain proof verification" beyond a single hackathon deployment. This ADR does not lift that prohibition — it scopes automated creation and settlement to this devnet deployment specifically, for the same reason ADR 0012 accepts an unvalidated lock-time buffer: devnet tokens have no real value, so the blast radius of a bad automated decision here is cosmetic, not financial.

## Decision

- **`contests` table** (`packages/db/migrations/0008_contests.sql`) is the new source of truth for per-fixture contest state (`PENDING_CREATION | CREATED | LOCKED | SETTLED`), replacing the single-contest env-var/hardcoded-literal assumption everywhere it existed: `lock-team`, `entry-transaction`, `live`, `share-card` routes, and `ReplayBuilder.tsx`'s entry-panel gate all now resolve contest state per fixture from this table instead of from `NEXT_PUBLIC_FANTASY_CONTEST_ID`/`FIXTURE_ID`/`LOCK_TS`.
- **`autoCreateContests()`** (`apps/worker/src/contest-lifecycle.ts`) opens a devnet contest for every fixture within a 7-day window that doesn't have one yet, reusing the exact PDA derivation, instruction layout, and simulate-before-send / verify-after-send safety pattern already proven in `preflight-fantasy-contest.mjs` — not reconstructed from scratch.
- **`checkAndSettleContests()`** watches for TxLINE's `game_finalised` action (normalized as `MATCH_FINALIZED`), confirms the ledger has zero unresolved scoring actions, then publishes settlement using the same simulate/send/verify pattern from `preflight-fantasy-settlement.mjs`, generalized from that script's hardcoded "exactly 3 payouts" to the real `buildSettlementPayouts` (`packages/solana-client/src/settlement.ts`), which handles any N.
- **Payout curve** (product decision, not an implementation detail): top-3 split, 50/30/20 of the vault; 100% for 1 entrant, 70/30 for 2. Named as `PAYOUT_CURVE` in `contest-lifecycle.ts` — change there, not by editing settlement math inline.
- **Credential custody**: the authority keypair is loaded from a file (`SOLANA_KEYPAIR_PATH`, same convention as both preflight scripts), read by the worker process at call time. This is the same credential as today's manual process, but it must now be continuously present on whatever host runs the worker instead of only present when a human runs a script by hand. That is the real security-posture change this ADR accepts, in exchange for zero manual steps.
- **Kill switch**: every exported function in `contest-lifecycle.ts` is a true no-op unless `FANTASY_AUTO_LIFECYCLE_ENABLED=true`. This is the fastest path to disabling automation without a code revert.
- **`contest-lifecycle.ts` does not import Solana instruction builders from `frontend/lib/contest-transaction.ts`, even though that file already had `buildCreateContestInstruction`/`buildPublishSettlementInstruction`.** It carries its own copies. Found by testing against real devnet, not by inspection: pnpm gives `frontend/` and `apps/worker/` separate physical installs of `@solana/web3.js` (confirmed via `readlink` on both — different `.pnpm` store paths despite identical version ranges). A `Transaction` built with one copy fails an internal `instanceof` check inside a `Connection` built with the other, surfacing as an opaque `Cannot read properties of undefined (reading 'numRequiredSignatures')` at simulate time — no type error, since TypeScript only checks structural shape, not physical module identity. `tsc --noEmit` passed cleanly on the broken version; only a real `simulateTransaction` call against devnet caught it. Any future code sharing Solana object construction across these two workspace packages will hit the same failure mode.

## Alternatives

Automate only contest creation, keep settlement manual (lower trust exposure, since settlement is what actually moves funds). Rejected — user explicitly asked for full automation including settlement. Noted here as the safer half-step if the auto-settler misbehaves in practice; disabling it alone (versus the whole kill switch) would require a small follow-up change to split the switch in two.

## Consequences

- Same devnet-only, no-real-value justification as ADR 0012. Not valid reasoning for a mainnet or otherwise-real-value deployment.
- **SSE stream scope is unverified.** `consumeScoreStream()` now assumes TxLINE's `/api/scores/stream` broadcasts all subscribed fixtures on one connection and routes messages to the right contest by `FixtureId`. If TxLINE instead requires one connection per fixture, live scoring for any fixture beyond the first will silently receive no events. This cannot be confirmed from static code and needs verification against real multi-fixture traffic once deployed.
- **Settlement's reconciliation check may never fire for some live fixtures.** `checkAndSettleContests()` confirms the ledger is reconciled by running `buildHistoricalReplayReadModel` over `raw_provider_events`, which requires a confirmed `lineups` action in that table. Live fixtures load their lineup via a separate REST snapshot (`apps/worker/src/lineup.ts`, `loadFixtureLineup`) that is not itself persisted into `raw_provider_events` — only whatever TxLINE re-broadcasts over the SSE stream lands there. If TxLINE never re-broadcasts the lineup action live, this check fails closed forever for that fixture (it simply never auto-settles, safely — no incorrect payout, just no automatic one) and the existing manual `preflight-fantasy-settlement.mjs` script remains the fallback. This needs the same live verification as the SSE stream scope item above.
- The manual preflight scripts still work exactly as before and remain the fallback path for both creation and settlement if automation is disabled or a specific fixture's automation stalls on either open item above. They do not currently write to the new `contests` table, so a manually-created contest needs a matching manual `insert into contests` for the worker's live-scoring bootstrap to pick it up.

## Migration

A production or mainnet port of this feature needs a new ADR that adds multi-signer settlement publication and/or on-chain proof verification before reusing this automated-authority-key design, per ADR 0008's own migration clause. The two open verification items above (SSE stream scope, live-fixture lineup persistence) should be resolved and logged here, not left as permanent unknowns, once this has run against real live TxLINE fixtures.
